## Draft

The experiments are designed to test whether verifier-grounded reasoning provides information beyond provider completion and fluent explanations. We ask three questions in the current pilot: whether LLM agents produce schema-valid verifiable reasoning traces, whether candidate constraints reduce verifier-visible failures, and whether verifier feedback improves reasoning reliability on traces that are eligible for revision.

Figure 1 summarizes the experiment pipeline: a controlled Guandan decision point is converted into a structured LLM reasoning trace, checked by rule-grounded verifier labels, revised using verifier feedback, and finally analyzed through paired attribution on the eligible revision subset. The figure is sourced from `docs/research/figures/figure-1-verifier-pipeline.svg` and the paired attribution artifact, so it is a visual summary of the current implementation rather than an illustrative placeholder. [Evidence: `docs/research/figures/figure-1-verifier-pipeline.svg`; `docs/research/experiments/pilot-verifier-attribution/verifier-attribution.json`.]

The benchmark consists of Guandan decision points generated from controlled `GameSession` states. Each decision point includes public history, acting player, team identity, hand counts, current table lead, legal candidate actions, inference summaries, and strategic scenario tags. The current full-evaluation artifact contains 500 schema-valid decision points under `docs/research/experiments/full-e1`, with manifest metadata explicitly marking the construction mode as `controlled_game_session_states`. [Evidence: `docs/research/experiments/full-e1/manifest.json`.]

The pilot implementation exports 50 schema-valid decision points covering the same five scenario tags: `lead_opening`, `follow_beat_or_pass`, `partner_near_finish`, `opponent_near_finish`, and `endgame_race`. The pilot is generated by `server/src/research/exportPilotDatasetCli.ts` and stored under `docs/research/experiments/pilot-e1`; it remains the cost-controlled harness-validation split, while `full-e1` is the current local full-evaluation split awaiting LLM-condition runs. [Evidence: `docs/research/experiments/pilot-e1/manifest.json`; `docs/research/experiments/full-e1/manifest.json`.]

We compare six artifact rows in the pilot table. Two deterministic baselines, `heuristic-legal-first` and `strategic-heuristic`, validate the local harness and verifier. The plain LLM receives the game state and outputs a reasoning trace and action. The candidate-constrained LLM is additionally shown the verifier-provided legal candidate list. The ToM-prompted LLM receives an explicit checklist for partner intent, opponent intent, and counterfactual candidate comparison under the same trace schema. The verifier-revision condition receives the candidate trace and verifier feedback, then produces a revised trace.

Current baseline artifacts include 50 `heuristic-legal-first` traces and 50 `strategic-heuristic` traces, each paired with hard-verifier outputs. Both baseline trace sets validate against `reasoning-trace.schema.json`, and both verifier result sets validate against `verifier-result.schema.json`. These artifacts establish reproducibility of the pipeline but do not replace LLM experiments.

The current pilot metrics summary is generated from experiment artifacts at `docs/research/experiments/pilot-metrics-summary`. The two deterministic baselines parse 50/50 traces and have zero hard failures, confirming that the schemas, verifier, and table generation path are internally consistent. The plain LLM condition has complete provider outputs for 50/50 prompts but only 26 parsed traces, with 24 schema parse failures and 26 hard verifier failures. The candidate-constrained condition also has complete provider outputs for 50/50 prompts, improves parse yield to 32 traces, but still has 35 hard verifier failures. The ToM-prompted condition has complete provider outputs for 50/50 prompts, parses 36 traces, and has 1 hard verifier failure among parsed traces. The verifier-revision condition is generated only for the 32 parsed candidate traces; it parses 32/32 revised traces and reduces hard failures to 10.

The ToM error analysis shows that prompt-level theory-of-mind reasoning mainly changes the failure surface rather than eliminating structured-output risk. Among the 14 ToM parse failures, 11 contain useful reasoning inside a nested `reasoning` object but do not expose the required top-level trace fields, 2 miss required trace fields under other aliases, and 1 is a tool-call-like output. Among the 36 parsed ToM traces, the only hard verifier failure is a hidden-information assertion. This supports a narrower interpretation: ToM prompting improves verifier-visible reasoning among parsed traces, but a schema-constrained decoding or repair layer is still needed for end-to-end reliability. [Evidence: `docs/research/experiments/pilot-e7-tom-failure-analysis/tom-failure-analysis.json`; `docs/research/experiments/pilot-e7-tom-prompted-results/metrics.json`.]

A deterministic schema-repair ablation tests that interpretation without changing the model's selected action. The repair layer preserves `selectedActionId`, passes through already valid traces, and reconstructs missing trace fields only from the raw ToM output plus public evidence ids. It raises ToM parse yield from 36/50 to 49/50 by repairing 13 nested or aliased outputs, while leaving the remaining tool-call-like output unrepaired and retaining 1 hard verifier failure after repair. This result supports the claim that most ToM failures in this pilot are schema-interface failures, but it does not remove the need for hidden-information verification. [Evidence: `docs/research/experiments/pilot-e8-tom-schema-repair-results/metrics.json`; `docs/research/experiments/pilot-e8-tom-schema-repair-results/schema-repair-report.json`.]

The paper-facing flow figure makes this denominator accounting explicit: 50 ToM provider outputs split into 36 raw schema-valid traces and 14 raw schema failures; schema repair recovers 13 of those failures and leaves 1 unrepaired output, yielding 49 verifier-eligible traces with 1 hard verifier failure. [Evidence: `docs/research/figures/figure-2-tom-schema-repair-flow.svg`; `docs/research/figures/figure-2-tom-schema-repair-flow.md`.]

The primary reasoning metrics are legal action rate, public-history consistency, hidden-information discipline, reasoning-action consistency, verifier warning rate, and correction success rate. The primary team-decision metrics are partner support success, opponent suppression success, bomb efficiency, pass regret, and team score proxy. We report these separately to avoid conflating verifiable reasoning with strategic optimality.

The main comparison shows that verifier revision reduces verifier-visible failures on the parsed candidate subset. Failure burden, defined as `fail + unknown`, drops most strongly for public-history consistency, from 29 before revision to 9 after revision. Hidden-information burden drops from 6 to 1, partner consistency burden from 29 to 26, opponent consistency burden from 22 to 20, reasoning-action burden from 1 to 0, and team-objective burden from 19 to 18. Legal-action and table-beating burdens are already zero in the candidate traces and remain zero after revision.

To address whether the revision effect is concentrated in verifier-grounded reasoning labels rather than only output formatting, we add a paired verifier-attribution analysis over the same 32 eligible candidate traces. The analysis reports hard failures dropping from 35 to 10, with a paired bootstrap 95% confidence interval of [-32, -18] for the hard-failure-count delta. At the decision level, hard-failure presence drops from 29 to 9 decisions, with 20 before-only failures and 0 after-only failures under a McNemar exact test. Component attribution shows that 80% of the hard-failure-count drop comes from public-history consistency and 20% from hidden-information discipline. The same artifact includes a qualitative case pack covering a public-history repair, a hidden-information repair, a remaining hard failure after revision, and a parse failure outside the revision subset. [Evidence: `docs/research/experiments/pilot-verifier-attribution/verifier-attribution.json`; `docs/research/experiments/pilot-verifier-attribution/verifier-attribution.md`.]

Ablations are not yet executed. The current repository includes an ablation table scaffold for removing hidden-information, partner, opponent, reasoning-action, and legal-action checks, but the rows remain `missing_metrics` and should not be interpreted as results.

The qualitative case pack illustrates what the paired attribution numbers mean operationally. In `pilot-e1-002-turn-1-player-0`, the candidate trace selected the same legal low-single action before and after revision, but its original partner evidence used natural-language claims not present as public evidence ids and omitted the public `partner_near_finish` tag. The revised trace keeps the action and rationale stable while replacing unsupported evidence with a public event id and explicitly hedging the partner's hidden cards; the public-history label changes from fail to pass and the partner-consistency warning disappears. In `pilot-e1-013-turn-1-player-0`, the first trace combines an unsupported public-evidence citation with a hidden-information assertion about opponents' cards. The revision keeps the selected action, rewrites the rationale, and changes the opponent belief from asserted holdings to public hand-count reasoning; both public-history and hidden-information labels change from fail to pass. In `pilot-e1-004-turn-1-player-0`, revision does not repair the public-history failure: the revised trace still cites a prose description rather than a valid public evidence id, so the verifier continues to emit `UNKNOWN_PUBLIC_EVIDENCE`. Finally, `pilot-e1-000-turn-1-player-0` shows the end-to-end parseability boundary: the raw candidate output contains only a compact action and free-form reasoning field, so it does not match the required structured trace schema and is excluded from the 32-trace revision subset. [Evidence: `docs/research/experiments/pilot-verifier-attribution/verifier-attribution.json`; raw and parsed trace files under `docs/research/experiments/pilot-e5-candidate-constrained-results` and `docs/research/experiments/pilot-e6-verifier-revision-results`.]
