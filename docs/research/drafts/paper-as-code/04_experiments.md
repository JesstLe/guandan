## Draft

The experiments are designed to test whether verifier-grounded reasoning provides information and improvements beyond outcome-only evaluation. We ask three questions: whether LLM agents produce verifiable reasoning, whether candidate constraints and ToM prompting reduce failures, and whether verifier feedback improves reasoning reliability without degrading team-decision quality.

The benchmark consists of Guandan decision points generated from controlled `GameSession` states. Each decision point includes public history, acting player, team identity, hand counts, current table lead, legal candidate actions, inference summaries, and strategic scenario tags. The current full-evaluation artifact contains 500 schema-valid decision points under `docs/research/experiments/full-e1`, with manifest metadata explicitly marking the construction mode as `controlled_game_session_states`. [Evidence: `docs/research/experiments/full-e1/manifest.json`.]

The pilot implementation exports 50 schema-valid decision points covering the same five scenario tags: `lead_opening`, `follow_beat_or_pass`, `partner_near_finish`, `opponent_near_finish`, and `endgame_race`. The pilot is generated by `server/src/research/exportPilotDatasetCli.ts` and stored under `docs/research/experiments/pilot-e1`; it remains the cost-controlled harness-validation split, while `full-e1` is the current local full-evaluation split awaiting LLM-condition runs. [Evidence: `docs/research/experiments/pilot-e1/manifest.json`; `docs/research/experiments/full-e1/manifest.json`.]

We compare five planned agent conditions. The deterministic strategic heuristic uses public scenario tags and legal candidates to select a transparent non-LLM action. The plain LLM receives the game state and outputs a reasoning trace and action. The candidate-constrained LLM chooses among legal actions. The ToM-prompted LLM explicitly reasons about partner intent and opponent threat. The verifier-in-the-loop LLM receives verifier feedback and revises before final action.

Current baseline artifacts include 50 `heuristic-legal-first` traces and 50 `strategic-heuristic` traces, each paired with hard-verifier outputs. Both baseline trace sets validate against `reasoning-trace.schema.json`, and both verifier result sets validate against `verifier-result.schema.json`. These artifacts establish reproducibility of the pipeline but do not replace LLM experiments.

The current pilot metrics summary is generated from experiment artifacts at `docs/research/experiments/pilot-metrics-summary`. It contains two deterministic baseline rows with verifier metrics and three LLM-condition rows marked `missing_raw_outputs`: plain, candidate-constrained, and verifier-revision. These missing LLM rows are not experimental results; they are machine-readable evidence that the prompt/batch/revision pipeline is waiting for real model outputs.

The primary reasoning metrics are legal action rate, public-history consistency, hidden-information discipline, reasoning-action consistency, verifier warning rate, and correction success rate. The primary team-decision metrics are partner support success, opponent suppression success, bomb efficiency, pass regret, and team score proxy. We report these separately to avoid conflating verifiable reasoning with strategic optimality.

The main comparison tests whether the verifier-in-the-loop condition reduces failures relative to plain and candidate-constrained prompting. A positive result would show lower hard-failure rates and higher reasoning-action consistency, while maintaining or improving team-decision metrics. [NEED_EXPERIMENT]

Ablations remove verifier components one at a time. We test variants without hidden-information checks, partner consistency checks, opponent consistency checks, reasoning-action consistency checks, and legal-action checks. These ablations identify which verifier components contribute to reliability improvements. [NEED_EXPERIMENT]

Qualitative case studies will show failures that outcome metrics miss. Planned cases include a legal winning action with invalid hidden-information reasoning, a fluent explanation that contradicts the selected action, a ToM prompt that hallucinates partner intent, and a verifier correction that changes both reasoning and action. [NEED_EXPERIMENT]
