## Draft

The experiments are designed to test whether verifier-grounded reasoning provides information beyond provider completion and fluent explanations. We ask three questions in the current pilot: whether LLM agents produce schema-valid verifiable reasoning traces, whether candidate constraints reduce verifier-visible failures, and whether verifier feedback improves reasoning reliability on traces that are eligible for revision.

The benchmark consists of Guandan decision points generated from controlled `GameSession` states. Each decision point includes public history, acting player, team identity, hand counts, current table lead, legal candidate actions, inference summaries, and strategic scenario tags. The current full-evaluation artifact contains 500 schema-valid decision points under `docs/research/experiments/full-e1`, with manifest metadata explicitly marking the construction mode as `controlled_game_session_states`. [Evidence: `docs/research/experiments/full-e1/manifest.json`.]

The pilot implementation exports 50 schema-valid decision points covering the same five scenario tags: `lead_opening`, `follow_beat_or_pass`, `partner_near_finish`, `opponent_near_finish`, and `endgame_race`. The pilot is generated by `server/src/research/exportPilotDatasetCli.ts` and stored under `docs/research/experiments/pilot-e1`; it remains the cost-controlled harness-validation split, while `full-e1` is the current local full-evaluation split awaiting LLM-condition runs. [Evidence: `docs/research/experiments/pilot-e1/manifest.json`; `docs/research/experiments/full-e1/manifest.json`.]

We compare five artifact rows in the pilot table. Two deterministic baselines, `heuristic-legal-first` and `strategic-heuristic`, validate the local harness and verifier. The plain LLM receives the game state and outputs a reasoning trace and action. The candidate-constrained LLM is additionally shown the verifier-provided legal candidate list. The verifier-revision condition receives the candidate trace and verifier feedback, then produces a revised trace. A ToM-prompted LLM condition remains outside the current pilot and is treated as future work.

Current baseline artifacts include 50 `heuristic-legal-first` traces and 50 `strategic-heuristic` traces, each paired with hard-verifier outputs. Both baseline trace sets validate against `reasoning-trace.schema.json`, and both verifier result sets validate against `verifier-result.schema.json`. These artifacts establish reproducibility of the pipeline but do not replace LLM experiments.

The current pilot metrics summary is generated from experiment artifacts at `docs/research/experiments/pilot-metrics-summary`. The two deterministic baselines parse 50/50 traces and have zero hard failures, confirming that the schemas, verifier, and table generation path are internally consistent. The plain LLM condition has complete provider outputs for 50/50 prompts but only 26 parsed traces, with 24 schema parse failures and 26 hard verifier failures. The candidate-constrained condition also has complete provider outputs for 50/50 prompts, improves parse yield to 32 traces, but still has 35 hard verifier failures. The verifier-revision condition is generated only for the 32 parsed candidate traces; it parses 32/32 revised traces and reduces hard failures to 10.

The primary reasoning metrics are legal action rate, public-history consistency, hidden-information discipline, reasoning-action consistency, verifier warning rate, and correction success rate. The primary team-decision metrics are partner support success, opponent suppression success, bomb efficiency, pass regret, and team score proxy. We report these separately to avoid conflating verifiable reasoning with strategic optimality.

The main comparison shows that verifier revision reduces verifier-visible failures on the parsed candidate subset. Failure burden, defined as `fail + unknown`, drops most strongly for public-history consistency, from 29 before revision to 9 after revision. Hidden-information burden drops from 6 to 1, partner consistency burden from 29 to 26, opponent consistency burden from 22 to 20, reasoning-action burden from 1 to 0, and team-objective burden from 19 to 18. Legal-action and table-beating burdens are already zero in the candidate traces and remain zero after revision.

Ablations are not yet executed. The current repository includes an ablation table scaffold for removing hidden-information, partner, opponent, reasoning-action, and legal-action checks, but the rows remain `missing_metrics` and should not be interpreted as results.

Qualitative case-study writing is also future work. The pilot artifacts already contain raw traces and verifier outputs suitable for selecting cases, especially schema-valid traces with public-history or hidden-information failures and revised traces where verifier feedback repairs a hard failure.
