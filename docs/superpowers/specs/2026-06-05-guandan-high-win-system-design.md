# 掼蛋高胜率辅助系统设计

## 1. 目标

把当前掼蛋助手从“能记录牌局并请求 AI 建议”升级为“规则可靠、候选出牌可计算、建议可解释、整局可复盘”的高胜率辅助系统。

这里的“必胜”定义为工程目标而不是承诺每局必赢：系统要在隐藏信息条件下持续提高决策质量，减少规则错误、录入错误、节奏错误、炸弹误用和队友配合失误。

## 2. 范围

本阶段覆盖四条能力线：

1. 规则和牌型强固化
2. 出牌建议增强
3. 复盘训练系统
4. 界面录入体验优化

实施顺序固定为 A -> B -> C -> D。每个阶段都要能独立验收，避免把规则、策略、复盘和 UI 混成一个不可测试的大改动。

## 3. 非目标

- 不读取或修改游戏进程内存。
- 不做屏幕识别或自动点击。
- 不承诺数学意义上的每局必赢。
- 不把 LLM 作为唯一决策来源。
- 不在同一阶段重写整个前后端架构。

## 4. 当前项目基础

仓库已经具备这些模块：

- React 前端：牌面展示、选牌器、玩家面板、建议面板、出牌历史等组件。
- Node 服务端：WebSocket/API、牌局会话、牌池、时间线、推断、进贡、规则校验。
- Shared 类型：牌、牌型、玩家、轮次、事件等共享模型。
- AI 层：结构化牌局状态、prompt 构建、响应解析和历史压缩。
- 测试基础：shared、server、client 都已有 Vitest 配置和多组测试。

这些模块适合继续演进，不需要另起炉灶。

## 5. 总体架构

系统分成五个核心层：

```text
前端录入/展示
  -> 牌局事件 API
  -> 规则和状态引擎
  -> 策略候选和评分引擎
  -> AI 解释和复盘层
```

规则和状态引擎负责“什么是合法、当前状态是什么”。策略引擎负责“有哪些选择、每个选择有多好”。AI 层负责“把推荐理由解释清楚、复盘时总结人类可理解的教训”。

## 6. A: 规则和牌型强固化

### 6.1 目标

让规则引擎成为所有后续策略的可信地基。任何出牌建议、复盘判断、UI 校验都必须依赖同一套纯规则函数。

### 6.2 核心能力

- 牌型识别：单牌、对子、三带、顺子、连对、钢板、飞机、飞机带翅膀、炸弹、同花顺、天王炸弹。
- 逢人配：红桃主牌可替代目标牌，并保留替代信息。
- 牌型比较：同类型比较、主牌优先、炸弹排序、同花顺与炸弹关系、天王炸弹最大。
- 局部规则开关：是否允许三带一、是否启用进贡、当前主牌等级。
- 牌局守恒：108 张牌、各家手牌数、已出牌、归档牌状态一致。

### 6.3 模块边界

- `server/src/engine/combinationDetector.ts` 只负责识别牌型。
- `server/src/engine/combinationCompare.ts` 只负责比较牌型。
- `server/src/validator/*` 负责输入合法性和状态合法性。
- `shared/src/types.ts` 提供稳定类型，不承载复杂业务逻辑。

### 6.4 验收标准

- 规则相关纯函数都有单元测试。
- 每个牌型至少包含正常、非法、边界、逢人配四类样例。
- 炸弹排序和同花顺排序有专门测试。
- 错误输入返回结构化错误，不只返回字符串。

## 7. B: 出牌建议增强

### 7.1 目标

系统能基于当前手牌和桌面状态，生成所有合法候选出法，排序并解释推荐原因。

### 7.2 候选生成器

新增一个独立策略模块：

```text
server/src/strategy/
  moveGenerator.ts
  moveScorer.ts
  moveExplainer.ts
  strategyTypes.ts
```

`moveGenerator` 输入：

- 我的手牌
- 当前桌面主牌型
- 当前主牌等级
- 规则开关
- 队友和对手剩余手牌数

输出：

- 可出的候选组合
- 过牌候选
- 每个候选的基础元数据，例如是否拆炸、是否拆顺、是否消耗逢人配、是否保留牌权

### 7.3 启发式评分

第一版使用可解释的启发式评分，不直接上复杂模型：

- 合法性分：非法为不可选。
- 出牌效率：减少手牌张数、减少散牌。
- 结构保护：避免拆炸、拆顺、拆连对、浪费逢人配。
- 牌权价值：当前是否需要抢牌权。
- 队友配合：队友快出完时优先放队友，敌人快出完时优先压制。
- 炸弹代价：炸弹用于终结、救急、断敌人时加分，普通抢小牌权时扣分。
- 风险控制：未知大牌和剩余炸弹越多，冒险出法扣分。

### 7.4 AI 角色

AI 不直接决定最终出法。服务端先计算候选和评分，再把 Top N 候选、局面摘要和评分理由交给 AI。AI 输出：

- 推荐出法
- 推荐原因
- 次优方案
- 风险提示
- 是否建议过牌

如果 AI 输出和策略评分冲突，服务端保留策略评分，并在 UI 中标记“AI 解释与规则评分不一致”。

### 7.5 验收标准

- 给定固定手牌和桌面牌，候选集合可快照测试。
- 评分结果稳定，Top 3 有明确理由。
- 不出现非法推荐。
- 推荐中明确标记消耗炸弹、逢人配、拆牌等代价。

## 8. C: 复盘训练系统

### 8.1 目标

每局结束后可以回放完整牌局，并找到关键决策点。

### 8.2 时间线模型

现有 timeline 能作为基础继续扩展。每个事件要包含：

- 操作玩家
- 出牌或过牌
- 出牌前后手牌数
- 当前牌权
- 系统当时 Top N 建议
- 用户实际选择
- 差异原因

### 8.3 复盘分析

复盘阶段重点标记：

- 错过牌权：明明有高价值接牌机会却过牌。
- 过早炸弹：炸弹收益低且破坏后续结构。
- 未保护队友：队友临门时没有放牌或转牌权。
- 未压敌人：敌方剩牌少时没有及时阻断。
- 拆牌损失：拆掉高价值组合导致后续手牌破碎。
- 录入异常：牌数守恒或牌池状态不一致。

### 8.4 验收标准

- 一局牌能导出为可回放 timeline。
- 每个关键节点能显示“当时系统建议”和“实际选择”。
- 复盘结论可由规则/评分数据支撑，不只是 AI 主观总结。

## 9. D: 界面录入体验优化

### 9.1 目标

让用户在真实对局中快速、低错误地录入牌局，并清楚看到系统建议和风险。

### 9.2 录入优化

- 选牌器区分“我的出牌”和“他人出牌”。
- 快捷录入显示每个点数剩余数量。
- 已出牌和不可选牌灰显。
- 撤回能回滚事件、牌池状态、推断状态和建议状态。
- 出牌前显示牌型识别结果，歧义牌型要求确认。

### 9.3 建议展示

建议面板显示：

- 首选出法
- 次选出法
- 建议过牌时的理由
- 消耗提示：炸弹、逢人配、拆牌
- 局势提示：队友剩牌、对手剩牌、当前牌权

### 9.4 复盘界面

复盘界面显示：

- 时间线列表
- 每一步系统建议和实际选择
- 关键错误标签
- 局面摘要
- 最终训练建议

### 9.5 验收标准

- 常见录入流程不需要手动输入文本。
- 录错后可以一键撤回。
- 建议信息不遮挡主牌局视图。
- 移动端不作为本阶段重点，桌面端优先。

## 10. 数据流

### 10.1 实时建议

```text
用户录入事件
  -> 前端发送 WebSocket 消息
  -> 后端校验事件
  -> 更新 GameSession/CardPool/Timeline
  -> 生成合法候选
  -> 评分排序
  -> 构建 AI 解释上下文
  -> 返回建议结果
  -> 前端更新建议面板和历史记录
```

### 10.2 复盘

```text
牌局结束
  -> 冻结 timeline
  -> 重新计算每个决策点的候选和评分
  -> 对比用户实际选择
  -> 生成关键节点列表
  -> AI 总结训练建议
  -> 前端展示回放和复盘结果
```

## 11. 错误处理

- 输入非法：返回具体原因，例如数量错误、重复牌、牌型非法、不能压过上家。
- 状态不守恒：阻止继续建议，并提示用户回滚或校正。
- AI 失败：仍展示本地策略评分，不让建议系统完全失效。
- 评分冲突：保留所有候选，并标记推荐依据。
- WebSocket 断开：前端保留本地当前输入，不自动丢弃。

## 12. 测试策略

### 12.1 单元测试

- 牌型识别
- 牌型比较
- 牌池状态更新
- 输入校验
- 候选生成
- 候选评分
- 响应解析

### 12.2 集成测试

- 开局录手牌 -> 出牌 -> 过牌 -> 撤回
- 队友快出完时的推荐
- 对手快出完时的阻断
- 炸弹使用场景
- AI 失败时 fallback 建议

### 12.3 回归样例

把典型牌局做成 fixtures。每修一个规则或策略问题，就新增一个 fixture，防止后续回退。

## 13. 分阶段实施计划

### 阶段 A: 规则强固化

交付：

- 完整牌型测试矩阵
- 逢人配识别和比较修正
- 炸弹/同花顺/天王炸弹排序测试
- 状态守恒错误结构化

### 阶段 B: 候选出牌和评分

交付：

- `strategy` 模块
- 合法出法生成器
- 启发式评分器
- 建议解释器
- Top N 建议 API/WS 输出

### 阶段 C: 复盘训练

交付：

- timeline 决策快照
- 关键节点分析器
- 复盘摘要
- 训练建议输出

### 阶段 D: UI 体验

交付：

- 更快的选牌和撤回流程
- 建议面板升级
- 历史/复盘视图
- 错误提示和状态校正入口

## 14. 成功标准

系统完成后应满足：

- 不推荐非法牌。
- 对同一局面输出稳定候选和评分。
- 推荐原因能解释到规则和策略代价。
- 用户能复盘一整局并看到关键决策点。
- 真实对局中录入速度足以跟上人工操作。
